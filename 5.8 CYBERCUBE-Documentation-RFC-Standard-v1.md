CYBERCUBE Documentation & RFC Standard (v1)

Glossary

This glossary defines key terms used throughout the CYBERCUBE Documentation & RFC
Standard.

All definitions are normative unless stated otherwise.

A

ADR (Architecture Decision Record)

A document capturing a significant architectural decision.

Components:
- Context (why)
- Decision (what)
- Consequences (impact)
- Status (proposed/accepted/deprecated)

Audience

The intended readers of a document.

Categories:
- Engineers (implementers)
- Operators (maintainers)
- Leadership (decision-makers)
- Customers (end users)

Author

The primary creator of a document.

Responsibilities:
- Initial draft
- Incorporating feedback
- Maintaining accuracy
- Tracking status

Approver

A person authorized to approve a document.

Role varies by document type.

C

Champion

The person driving an RFC through the process.

Also known as: Author, Owner

Changelog

A record of changes to a document or system.

See also: Version History

Contributor

Anyone who provides input to a document.

Activities:
- Reviewing
- Commenting
- Suggesting edits

D

Deprecated

A status indicating a document is outdated and should not be used.

Lifecycle: Active â†’ Deprecated â†’ Archived

Document Type

A classification of documentation by purpose.

CYBERCUBE types:
- Standard
- Guide
- RFC
- ADR
- Runbook
- Reference
- Tutorial

Draft

An incomplete document under development.

Not yet approved or published.

E

Effective Date

The date a document becomes active/binding.

Required for: Standards, Policies

F

Feedback Period

A designated time for collecting comments on a document.

RFC minimum: 2 weeks

G

Guide

A document providing recommendations (not requirements).

Compliance: Suggested, not mandatory.

Also known as: Best Practice, How-To

H

Heading

A section title in a document.

Levels: H1 (title), H2 (section), H3 (subsection)

I

Informative

Content that provides context but is not binding.

Contrast with: Normative

L

Lifecycle

The stages a document passes through.

Stages:
- Draft
- Review
- Approved
- Active
- Deprecated
- Archived

M

Maintainer

The person responsible for keeping a document current.

May differ from author after transfer.

Markdown

The markup language for CYBERCUBE documentation.

Format: CommonMark with GFM extensions

Metadata

Structured information about a document.

Fields:
- Title
- Version
- Status
- Author
- Date
- Tags

N

Normative

Content that establishes requirements.

Keywords: MUST, SHALL, REQUIRED

Contrast with: Informative

O

Owner

The person accountable for a document.

Responsibilities:
- Accuracy
- Currency
- Access
- Retirement

P

Policy

A high-level document stating organizational positions.

Enforcement: Mandatory compliance.

Proposal

A document suggesting a change or new capability.

Synonym: RFC

R

Reference

A document providing detailed specifications.

Format: API docs, schemas, configurations

Review

The process of evaluating a document for quality and accuracy.

Types:
- Peer review
- Technical review
- Leadership review

Reviewer

A person who evaluates a document.

Provides feedback, may approve.

RFC (Request for Comments)

A proposal document for significant changes.

Process:
1. Draft proposal
2. Solicit feedback
3. Revise
4. Decide (accept/reject)

Runbook

A document providing step-by-step operational procedures.

Also known as: Playbook, SOP

S

Scope

The boundaries of what a document covers.

States: In scope, out of scope

Section

A major division of a document.

Numbered for standards, titled for guides.

Stakeholder

Anyone affected by a document's content.

Categories:
- Primary (directly affected)
- Secondary (indirectly affected)

Standard

A document establishing mandatory requirements.

Compliance: Required.

Keywords: MUST, SHALL, REQUIRED

Status

The current state of a document.

Values: Draft, Review, Approved, Active, Deprecated, Archived

Style Guide

A document defining writing conventions.

Covers: Tone, formatting, terminology

T

Template

A pre-structured document format.

Purpose: Consistency, completeness

TOC (Table of Contents)

An index of document sections.

Auto-generated for long documents.

V

Version

A unique identifier for a document revision.

Format: `v{major}.{minor}` or `v{major}.{minor}.{patch}`

Version History

A record of document versions and changes.

Location: End of document

0. Purpose & Design Principles

This standard defines how CYBERCUBE creates, manages, and governs documentation.
It establishes document types, naming conventions, the RFC process for proposing
changes, and ownership models.

Industry alignment:
- IETF RFC Process
- Amazon ADR Model
- Google Design Documents
- Stripe Documentation Standards

Design principles:

1. **Clarity** â€” Write for the reader, not the writer
2. **Consistency** â€” Same structure, same conventions
3. **Currency** â€” Documents reflect current state
4. **Discoverability** â€” Easy to find, well-organized
5. **Accountability** â€” Clear ownership
6. **Versioned** â€” Changes are tracked

This document does NOT define:
- Customer-facing documentation style
- Marketing content
- Legal documents

1.2 Document Types Defined

| Type | Purpose | Compliance | Audience | Examples |
|------|---------|------------|----------|----------|
| Standard | Establish requirements | Mandatory | Engineers, Ops | Naming Standard |
| Policy | State organizational positions | Mandatory | All | Security Policy |
| RFC | Propose significant changes | N/A (process) | Stakeholders | New auth system |
| ADR | Record architectural decisions | Informative | Engineers | Database choice |
| Guide | Provide recommendations | Suggested | Engineers | Code style |
| Runbook | Document procedures | Required when applicable | Ops | Incident response |
| Reference | Detail specifications | Informative | Engineers | API documentation |
| Tutorial | Teach a skill | Informative | Engineers | Getting started |

1.3 Standards vs Guides

| Aspect | Standard | Guide |
|--------|----------|-------|
| Compliance | MUST follow | SHOULD follow |
| Language | Normative (SHALL, MUST) | Advisory (recommend, consider) |
| Approval | Formal review | Team review |
| Exceptions | Documented + approved | At discretion |
| Updates | Versioned, announced | Continuous |

**Standard example:**
> "All public identifiers MUST use the CC-PID format: `{prefix}_{random}`"

**Guide example:**
> "Consider using descriptive variable names that indicate the data type"

1.4 RFC vs ADR

| Aspect | RFC | ADR |
|--------|-----|-----|
| Purpose | Propose change | Record decision |
| Timing | Before decision | After decision |
| Scope | Significant changes | Architectural choices |
| Process | Review + approval | Document + commit |
| Outcome | Accept/Reject | Record of why |

**RFC:** "Should we migrate from PostgreSQL to CockroachDB?"
**ADR:** "We chose PostgreSQL because..."

1.5 Normative Language

Standards use RFC 2119 keywords:

| Keyword | Meaning |
|---------|---------|
| MUST / SHALL / REQUIRED | Absolute requirement |
| MUST NOT / SHALL NOT | Absolute prohibition |
| SHOULD / RECOMMENDED | Strong recommendation |
| SHOULD NOT / NOT RECOMMENDED | Strong discouragement |
| MAY / OPTIONAL | Truly optional |

```markdown
<!-- Standard language -->
All API endpoints MUST return JSON responses.
Authentication tokens SHOULD be rotated every 24 hours.
Clients MAY cache responses for up to 5 minutes.

<!-- Guide language -->
We recommend using descriptive error messages.
Consider implementing retry logic for transient failures.
You might want to add logging for debugging purposes.
```

3. Naming & Organization

Consistent naming and organization enables discoverability.

3.1 Document Naming Convention

Format: `{Type}-{Name}-{Version}.md`

> **Note:** Standards within the CYBERCUBE catalog MAY use a numeric prefix
> (e.g., `5.8 CYBERCUBE-...`) to indicate catalog ordering. This prefix is
> a catalog convention and is separate from the document naming format below.

| Type | Prefix | Example |
|------|--------|---------|
| Standard | `CYBERCUBE-{Name}-Standard` | `CYBERCUBE-Naming-Identifier-Standard-v1.1.md` |
| Policy | `CYBERCUBE-{Name}-Policy` | `CYBERCUBE-Security-Policy-v1.md` |
| RFC | `RFC-{number}-{Name}` | `RFC-0042-New-Auth-System.md` |
| ADR | `ADR-{number}-{Name}` | `ADR-0015-PostgreSQL-Selection.md` |
| Guide | `{Name}-Guide` | `TypeScript-Style-Guide.md` |
| Runbook | `Runbook-{Name}` | `Runbook-Database-Failover.md` |
| Reference | `{Name}-Reference` | `API-v1-Reference.md` |

3.2 Numbering Systems

**RFC numbers:**
```
RFC-0001 through RFC-9999
Sequential assignment
Never reused
```

**ADR numbers:**
```
ADR-0001 through ADR-9999
Sequential assignment
Never reused
```

**Standard versions:**
```
v{major}.{minor}
v1.0, v1.1, v2.0
Major: Breaking changes
Minor: Additions/clarifications
```

4. RFC Process

RFCs propose significant changes requiring broad input.

4.1 When to Write an RFC

**RFC required for:**
- New services or major components
- Changes affecting multiple teams
- Breaking API changes
- Security model changes
- New technology adoption
- Significant process changes

**RFC not required for:**
- Bug fixes
- Minor features
- Refactoring
- Documentation updates
- Configuration changes

4.3 RFC Status Values

| Status | Description |
|--------|-------------|
| Draft | Under development by author |
| Review | Open for feedback |
| Accepted | Approved for implementation |
| Rejected | Not approved |
| Deferred | Postponed for future consideration |
| Superseded | Replaced by newer RFC |
| Withdrawn | Withdrawn by author |

4.4 RFC Process Steps

```markdown
## RFC Process

### 1. Draft (Author)
- [ ] Assign RFC number (sequential)
- [ ] Write proposal using RFC template
- [ ] Identify stakeholders and reviewers
- [ ] Create PR with RFC document

### 2. Review (2 weeks minimum)
- [ ] Announce RFC to stakeholders
- [ ] Collect feedback via PR comments
- [ ] Address questions and concerns
- [ ] Revise proposal as needed
- [ ] Mark as "Review" status

### 3. Decision Meeting
- [ ] Schedule decision meeting
- [ ] Invite key stakeholders
- [ ] Present final proposal
- [ ] Discuss outstanding concerns
- [ ] Make accept/reject decision

### 4. Finalize
- [ ] Update status (Accepted/Rejected)
- [ ] Document decision rationale
- [ ] Merge PR
- [ ] Announce decision
- [ ] Create implementation tasks (if accepted)
```

4.5 RFC Review Guidelines

**For reviewers:**
- Focus on the problem, not just the solution
- Suggest alternatives constructively
- Identify risks and edge cases
- Consider operational impact
- Evaluate against existing standards
- Provide actionable feedback

**Review questions:**
- Does this solve the stated problem?
- Are there simpler alternatives?
- What are the operational implications?
- How does this affect existing systems?
- What are the security implications?
- Is the implementation plan realistic?

5. ADR Process

ADRs capture the context and consequences of architectural decisions.

5.1 When to Write an ADR

**ADR required for:**
- Technology selection (database, framework)
- Architectural patterns adopted
- Significant trade-offs made
- Decisions with long-term impact
- Reversing previous decisions

**ADR not required for:**
- Obvious choices (following standards)
- Temporary solutions
- Minor implementation details

5.3 ADR Writing Tips

**Context section:**
- Describe the situation objectively
- Include constraints and requirements
- Note relevant background

**Decision section:**
- State the decision clearly
- Be specific and actionable
- Include what we will NOT do

**Consequences section:**
- List positive outcomes
- Acknowledge negative trade-offs
- Note neutral observations

```markdown
## Context

We need a primary database for user data. Requirements:
- ACID transactions
- Strong consistency
- Horizontal scaling potential
- Team familiarity
- Managed service available

Current options: PostgreSQL, MySQL, CockroachDB

## Decision

We will use **PostgreSQL** as our primary relational database.

We will NOT use:
- MySQL: Less robust JSON support
- CockroachDB: Overkill for current scale, higher cost

## Consequences

### Positive
- Team has deep PostgreSQL expertise
- Excellent JSON/JSONB support for flexible schemas
- Strong ecosystem (pgvector, PostGIS)
- Cloud Run/Cloud SQL integration

### Negative
- Single-region limitation initially
- Manual sharding if needed later
- No built-in multi-master

### Neutral
- Standard SQL, portable queries
- Well-documented operational procedures
```

6. Versioning

Documents are versioned to track changes and ensure clarity.

6.1 Version Format

**Standards and Policies:**
```
v{major}.{minor}

v1.0  - Initial release
v1.1  - Minor additions/clarifications
v2.0  - Breaking changes
```

**RFCs and ADRs:**
```
No version numbers
Document status changes instead
Superseded documents link to replacements
```

**Guides and Runbooks:**
```
Date-based: Last Updated: YYYY-MM-DD
Or version: v{major}.{minor}
```

6.4 Breaking Changes

Breaking changes to standards require:

1. **Announcement:** Notify affected teams 30+ days ahead
2. **Migration guide:** Document how to comply
3. **Grace period:** Allow time for migration
4. **Support:** Help teams transition

```markdown
## Breaking Changes in v2.0

### BC-1: New identifier format

**Previous:** `usr_abc123`
**New:** `usr_A1B2C3D4E5F6`

**Migration:**
1. Update ID generation to new format
2. Both formats accepted during transition (until 2026-06-01)
3. Run migration script for existing IDs

**Timeline:**
- 2026-01-17: v2.0 released
- 2026-03-17: New format required for new records
- 2026-06-01: Old format deprecated
```

8. Style Guide

Consistent style improves readability and professionalism.

8.1 Language and Tone

**Use:**
- Active voice
- Present tense
- Second person ("you") for instructions
- Direct, concise sentences
- Technical terms consistently

**Avoid:**
- Passive voice (unless necessary)
- Future tense (use present)
- Jargon without definition
- Unnecessarily long sentences
- Ambiguous pronouns

```markdown
<!-- Good -->
To create a user, call the POST /users endpoint.
The server returns a 201 status code on success.

<!-- Bad -->
A user can be created by calling the POST /users endpoint.
A 201 status code will be returned by the server when the operation has been completed successfully.
```

9.4 Automated Checks

```yaml
# .github/workflows/docs.yml
name: Documentation

on:
  pull_request:
    paths:
      - 'docs/**'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Lint Markdown
        uses: DavidAnson/markdownlint-cli2-action@v14
        with:
          globs: 'docs/**/*.md'
      
      - name: Check links
        uses: lycheeverse/lychee-action@v1
        with:
          args: --verbose docs/**/*.md
      
      - name: Spell check
        uses: streetsidesoftware/cspell-action@v5
        with:
          files: docs/**/*.md
```
